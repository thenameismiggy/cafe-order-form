# UX Guards — Design Spec
**Date:** 2026-03-22
**Status:** Approved

## Overview

Two defensive improvements: a maximum quantity cap of 10 on the order panel stepper, and a visible error banner when localStorage writes fail.

Note: The originally planned `seedMenuIfNeeded` ID counter fix is not needed — `saveIdCounter(DEFAULT_MENU.length)` is already called during seeding at line 318 of the current codebase.

---

## Change 1: Stepper Maximum Cap (10)

The `+` stepper button in the order panel currently has no upper bound.

**New behaviour:** The `+` button is disabled (via the `disabled` attribute) when the current quantity for an item reaches 10. This mirrors the existing pattern where `-` is disabled at 0. The existing 0-floor behaviour for the `-` button is unchanged by this spec.

The cap is enforced in two places:

**In `renderOrderPanel()`** where stepper buttons are rendered (line 553), add the `disabled` condition to the `+` button:

```html
<button class="stepper-plus" data-item-id="${item.id}" ${qty >= 10 ? 'disabled' : ''} aria-label="Increase">+</button>
```

**In the existing `querySelectorAll('.stepper-plus').forEach` callback** (lines 578–583), add an early return before incrementing:

```js
panelEl.querySelectorAll('.stepper-plus').forEach(btn => {
  btn.addEventListener('click', () => {
    const id = parseInt(btn.dataset.itemId, 10);
    if ((appState.panelQtys[id] || 0) >= 10) return;
    appState.panelQtys[id] = (appState.panelQtys[id] || 0) + 1;
    renderOrderPanel();
  });
});
```

The JS guard is included as defense-in-depth: native `disabled` prevents the click event in standard browsers, but the guard protects against programmatic invocation (e.g., `btn.click()` from the console) and remains correct when the Code Quality spec migrates steppers to event delegation (where the handler runs before checking element state).

**Existing orders with quantities above 10:** Not possible in normal use since the cap is enforced on input. No clamping is applied on load — if a quantity above 10 exists in persisted state (e.g., data from before this change), it is displayed as-is and the `+` button is disabled. No migration needed.

## Change 2: localStorage Error Banner

**Scope of this spec:** This spec wraps the two currently-existing localStorage write functions. The two new functions introduced by other specs (`saveCategories()` from Code Quality spec, `persistOrders()` from Persistence spec) are wrapped in try/catch within their own respective specs — each spec is responsible for hooking into `showStorageError()` when it introduces a new write path.

The following existing functions are wrapped in try/catch as part of this change:

- `saveMenu()` (line 287–289) — currently unwrapped
- `saveIdCounter()` (line 293–295) — currently unwrapped

Both call `showStorageError()` in their catch block.

**`showStorageError()`:**
- Checks for an existing banner via `document.getElementById('storage-error-banner')`
- If one is already present (visible), does nothing — no stacking
- If none is present, injects a dismissible banner as the **first child of `<body>`**, before `<header>`, in normal document flow (no `position: fixed`):

```html
<div id="storage-error-banner" style="background:#dc2626;color:#fff;padding:0.75rem 1rem;display:flex;justify-content:space-between;align-items:center;">
  <span>Storage error: your changes could not be saved. Please contact your developer.</span>
  <button onclick="this.parentElement.remove()" style="background:none;border:none;color:#fff;font-size:1.2rem;cursor:pointer;">×</button>
</div>
```

- The banner is inserted in normal document flow before `<header>` — this pushes page content down by approximately the banner height (~48px). This layout shift is acceptable: storage errors are rare and the banner must be clearly visible to staff.
- The `×` button removes the banner from the DOM via `this.parentElement.remove()`
- After dismissal, if another save subsequently fails, `showStorageError()` will find no existing banner and inject a new one — staff is always notified of new failures

The app continues working in-memory for the session — staff is not blocked from taking orders.

## Out of Scope

- Per-item quantity limits (different max per menu item)
- Retry logic for failed localStorage writes
- Offline detection or service worker caching
- Repair of existing menu data with conflicting IDs
